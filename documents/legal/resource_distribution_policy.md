# Политика разработки и распространения ресурсов

## Статус документа

Это внутренняя инженерная политика проекта, а не юридическое заключение. Она задаёт безопасный режим разработки до получения письменного подтверждения прав от правообладателей или профильного юриста.

## Модель распространения по умолчанию

Публичный репозиторий и сборки проекта содержат только:

- новый исходный код Flutter/Dart, Swift, Objective-C++ и C++;
- исходный код импортёра и валидаторов;
- документацию форматов, полученную наблюдением за данными;
- схемы собственных промежуточных форматов;
- тесты и синтетические фикстуры, созданные командой проекта;
- конфигурацию сборки и документацию пользователя.

Оригинальные игровые ресурсы пользователь извлекает локально из собственной законно приобретённой копии игры. Импортёр не скачивает, не поставляет и не публикует эти данные.

## Данные, запрещённые к размещению в Git и артефактах проекта

Без отдельного письменного разрешения нельзя добавлять:

- оригинальные исполняемые файлы, библиотеки и установщики;
- контейнеры `.KWN`, аудиофайлы `.RWS` и файл сохранения `AOXXL.sav`;
- извлечённые модели, текстуры, анимации, уровни, музыку, речь и звуковые эффекты;
- образы дисков, обложки и сопутствующие материалы оригинального издания;
- фрагменты декомпилированного исходного кода или дизассемблированную реализацию;
- сгенерированные пакеты ресурсов, из которых можно восстановить оригинальный контент.

Запрет распространяется на историю Git, release assets, CI-кеши, логи, тестовые отчёты и публичные хранилища сборок.

## Допустимые тестовые данные

В репозитории разрешены только:

1. полностью синтетические фикстуры без оригинального контента;
2. минимальные бинарные структуры, созданные командой для проверки собственного парсера;
3. данные с явно зафиксированной совместимой лицензией и указанием происхождения;
4. хеши, размеры и технические метаданные локальных файлов, если они не содержат сам контент.

Каждая несинтетическая фикстура должна иметь запись о происхождении, лицензии и разрешённом способе использования.

## Разделение рабочих данных

Локальная копия оригинальной игры и результаты импорта должны находиться вне репозитория. Инструменты принимают путь к ним через аргумент CLI или локальную переменную окружения. Значения путей и содержимое локального каталога не записываются в Git.

Рекомендуемое разделение:

```text
asterix/                 # публичный репозиторий нового кода
asterix-local-source/    # локальная копия исходных игровых данных
asterix-local-build/     # локально сгенерированные runtime-ресурсы
```

## Правила исследования форматов

- Документировать наблюдаемую структуру данных, входы, выходы и подтверждённые гипотезы.
- Не копировать реализацию оригинального executable в новый код.
- Для каждого вывода сохранять воспроизводимый тест или инструмент проверки.
- Не включать оригинальные байты в сообщения об ошибках, snapshots и CI-логи.
- Не публиковать инструмент обхода защиты, если такая защита будет обнаружена; работу остановить до отдельной юридической оценки.

## Контроль перед публикацией

Перед каждым публичным релизом необходимо:

1. проверить Git-историю и release artifacts на запрещённые файлы;
2. проверить лицензии зависимостей и всех тестовых данных;
3. убедиться, что приложение работает с локальным импортом и не содержит встроенных оригинальных ресурсов;
4. отдельно согласовать использование названия, персонажей, изображений, музыки и иных обозначений в названии продукта и маркетинговых материалах;
5. получить юридическую оценку для выбранных стран распространения.

## Условия изменения политики

Политика может быть ослаблена только на основании письменного разрешения, в котором явно перечислены правообладатель, разрешённые материалы, территории, способы распространения и срок действия. Ссылка на документ и область разрешения фиксируются здесь без публикации конфиденциальных данных.

До такого подтверждения действует модель «движок и импортёр без оригинальных ресурсов».
