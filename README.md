# Asterix & Obelix XXL macOS runtime

Оригинальные игровые ресурсы не входят в репозиторий. Перед первым запуском
соберите локальный ASTPAK из своей установленной копии игры:

```sh
./scripts/install_slice_assets.sh "/путь/к/AsterixXXL"
```

Скрипт установит пакет в
`~/Library/Application Support/AsterixXXL/gaul-stage-1.astpak`. После этого
`make run` и запуск «Новой игры» находят его автоматически. Если стандартного
пакета нет, приложение предлагает выбрать ранее собранный `.astpak` вручную.

Для локальной установки Triada в текущем окружении команда выглядит так:

```sh
./scripts/install_slice_assets.sh "$HOME/Downloads/Asterix & Obelix XXL (Triada)/prefix/drive_c/AsterixXXL"
```

Сборка требует закреплённый через FVM Flutter SDK. ASTPAK и производные игровые
данные должны оставаться вне Git.

## Почему ASTPAK генерируется отдельным скриптом

ASTPAK — это локальный runtime-пакет, а не исходный файл проекта. Он собирается
из ресурсов установленной копии Asterix & Obelix XXL и содержит подготовленные
для runtime модели, текстуры, анимации, collision-данные и аудио. Приложение
читает один проверяемый `.astpak`, не работая напрямую с исходными форматами
игры и россыпью промежуточных файлов.

Генерация вынесена в `scripts/build_slice_assets.sh`, чтобы отделить импорт
пользовательских данных от обычной сборки приложения. Скрипт выполняет весь
воспроизводимый pipeline: извлекает во временный каталог importer proof,
проверяет его и детерминированно упаковывает в ASTPAK. Благодаря этому один и
тот же процесс можно запустить для любой локальной установки игры, не храня
машинно-зависимые пути и результаты импорта в репозитории. Для повседневного
запуска `scripts/install_slice_assets.sh` вызывает этот pipeline и устанавливает
готовый пакет в стандартный каталог приложения.

## Почему ASTPAK не коммитится в Git

Хотя формат ASTPAK разработан в этом проекте, содержимое конкретного пакета
является производным от оригинальных игровых ресурсов. Из него можно получить
модели, текстуры, анимации и звук, права на распространение которых репозиторий
не предоставляет. Поэтому в Git хранятся только код импортёра, описание формата
и синтетические тестовые данные, а каждый разработчик создаёт ASTPAK локально из
своей законно полученной копии игры.

Такой подход также не раздувает историю Git большими бинарными файлами и не
создаёт устаревшие пакеты при изменении импортёра или схемы ASTPAK. Пакет всегда
можно пересобрать актуальной версией скрипта; локальные результаты сборки и
кеши остаются за пределами репозитория согласно
[`documents/legal/resource_distribution_policy.md`](documents/legal/resource_distribution_policy.md).
